# F2 - Schedule RA Job Interview

## Overview

F2 is the interview scheduling feature that allows students to respond to faculty interview requests for RA job applications. Students can accept proposed time slots, suggest alternative times, or cancel interviews within a 24-hour window.

**Feature Status**: ✅ Complete (with test data for development)  
**Database Integration**: Pending (works with test data)  
**Email Reminders**: Ready to enable (currently logs to console)

## User Workflows

### Student: View Pending Interviews

**URL**: `/interview/myInterviews`

Students access their interview list showing all pending, accepted, and cancelled interviews.

**Display Elements**:
- Faculty name
- Interview status (badge with color coding)
- Proposed time slots (for PENDING interviews)
- Accepted time (for ACCEPTED/CONFIRMED)
- Created date

**Status Badge Colors**:
- 🟠 Orange - PENDING (awaiting student response)
- 🟢 Green - ACCEPTED (slot confirmed)
- 🔵 Blue - COUNTERED (student proposed alternative)
- 🔷 Teal - CONFIRMED (both parties agreed)
- 🔴 Red - CANCELLED (interview cancelled)

### Student: Accept Proposed Slot

**URL**: `/interview/detail/{id}`

When viewing a PENDING interview, student selects one of the three faculty-proposed times.

**Steps**:
1. Click "View" on interview in list
2. See 3 proposed time slots as radio options
3. Select desired slot
4. Click "Accept Selected Slot"
5. Receive confirmation: "Interview slot accepted! You will receive a reminder before your interview."

**Backend**: Interview status changes to ACCEPTED, acceptedSlot is set

### Student: Propose Alternative Time

**URL**: `/interview/counter/{id}`

Student can propose a different time if none of the faculty's slots work.

**Steps**:
1. Click "Propose Alternative Time" button
2. Select date using calendar picker
3. Select time using time picker
4. (Optional) Add note about scheduling constraints or timezone
5. Click "Submit Counter-Offer"
6. Receive confirmation

**Backend**: Interview status changes to COUNTERED, counterSlot and counterNote stored

### Student: Cancel Interview

**URL**: `/interview/detail/{id}` (for ACCEPTED interviews)

Student can cancel an ACCEPTED interview, but only if 24+ hours remain before the scheduled time.

**Steps**:
1. Click "Cancel Interview" button (shown for ACCEPTED status)
2. Enter reason (optional)
3. Confirm cancellation
4. Receive confirmation or error if outside cancellation window

**Error Case**: "Cannot cancel: you may be outside the cancellation window (must cancel at least 24h before)."

**Backend**: Interview status changes to CANCELLED, cancelledBy="STUDENT", cancelReason stored

## Interview Reminders

### Automatic Reminders

The system automatically sends email reminders to students for confirmed interviews:

**24-Hour Reminder**:
- Sent approximately 24 hours before the interview
- Includes: Faculty name, interview time, location, meeting link

**1-Hour Reminder**:
- Sent approximately 1 hour before the interview
- Urgent: "Interview starting in 1 hour"

### Enabling Email Reminders

By default, reminders log to console. To enable actual email delivery:

1. Update `backend/conf/application.conf`:
```conf
email {
  enabled = true
  from    = "your-email@company.com"
  smtp {
    host     = "smtp.gmail.com"
    port     = 587
    auth     = true
    starttls = true
    user     = ${MAIL_USER}      # Set as environment variable
    password = ${MAIL_PASSWORD}  # Set as environment variable
  }
}
```

2. Set environment variables:
```bash
export MAIL_USER="your-email@gmail.com"
export MAIL_PASSWORD="your-app-password"  # Gmail app-specific password
```

3. Restart backend

### Reminder Status

Students can see reminder status on their interview detail page:
- ✓ "24-hour reminder sent at 2026-04-29T09:00:00"
- ⏳ "Waiting for 1-hour reminder..."

## API Endpoints

### Student API

All endpoints require authentication (session-based with Play Framework).

**List Student Interviews**:
```
GET /interview/byStudent/:studentId
Response: [ { InterviewRequest }, ... ]
```

**Get Interview Detail**:
```
GET /interview/detail/:id
Response: { InterviewRequest }
```

**Accept a Slot**:
```
POST /interview/accept/:id
Body: { "chosenSlot": "2026-05-10T14:00:00" }
Response: { InterviewRequest }
```

**Submit Counter-Offer**:
```
POST /interview/counter/:id
Body: {
  "counterSlot": "2026-05-15T10:00:00",
  "counterNote": "Afternoon works better"
}
Response: { InterviewRequest }
```

**Cancel Interview**:
```
POST /interview/cancel/:id
Body: { "reason": "Schedule conflict" }
Response: { success: boolean }
```

### Scheduler API (Backend Management)

**Mark 24h Reminder Sent**:
```
POST /interview/reminder24h/:id
Response: "24h reminder marked"
```

**Mark 1h Reminder Sent**:
```
POST /interview/reminder1h/:id
Response: "1h reminder marked"
```

**My Interviews (Frontend route)**:
```
GET /interview/myInterviews
Response: Renders interviewRequestList view for logged-in student
```

## Data Model

### InterviewRequest Fields

| Field | Type | Description |
|-------|------|-------------|
| id | Long | Interview request ID |
| raJobApplication | Relationship | RA job application |
| faculty | User | Faculty conducting interview |
| student | User | Student being interviewed |
| proposedSlot1/2/3 | String | ISO-8601 datetime strings for faculty-proposed times |
| acceptedSlot | String | Time slot student accepted (null if not accepted) |
| counterSlot | String | Alternative time student proposed (null if no counter-offer) |
| counterNote | String | Optional note with counter-offer |
| status | String | PENDING \| ACCEPTED \| COUNTERED \| CONFIRMED \| CANCELLED \| EXPIRED |
| location | String | Interview location (e.g., "Room 301", "Zoom") |
| meetingLink | String | URL for virtual interviews |
| notes | String | Faculty notes to student |
| reminder24hSentTime | String | Timestamp when 24h reminder was sent |
| reminder1hSentTime | String | Timestamp when 1h reminder was sent |
| cancelledBy | String | "STUDENT" \| "FACULTY" (who initiated cancellation) |
| cancelReason | String | Reason for cancellation |
| cancelledTime | String | When cancellation occurred |
| cancelWindowHours | Int | Hours before interview student can still cancel (default: 24) |
| isActive | String | "True" \| "False" (soft delete) |
| createTime | String | When interview request was created |
| updateTime | String | When interview request was last updated |

### Persistence Strategy

**Without Database**: Frontend controller caches test data in static HashMap that persists across requests during same session. Changes visible when navigating back and forth.

**With Database**: Backend service saves all changes to interview_request table. Frontend receives fresh data on each request.

## Troubleshooting

### Interviews not appearing in list
- Check auto-login is working: Should see "Test User" in session
- Frontend service may not be calling backend correctly
- Check browser console for API errors

### Reminders not sending
- `email.enabled = false` in config? Set to `true`
- Missing environment variables `MAIL_USER` or `MAIL_PASSWORD`?
- Check application logs: `tail -f backend/logs/application.log`

### Status not changing after accepting/cancelling
- Test data persists in memory, changes survive page reloads
- To reset, restart frontend server
- With real database, changes persist to disk
